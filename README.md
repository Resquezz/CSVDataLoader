# C# Test Assessment — CSV to SQL ETL

Опис
----------------
- CLI-програма на C# для ETL: зчитування CSV, очищення, видалення дублікатів і масова вставка в MS SQL таблицю.
- Імпортуються лише колонки: `tpep_pickup_datetime`, `tpep_dropoff_datetime`, `passenger_count`, `trip_distance`, `store_and_fwd_flag`, `PULocationID`, `DOLocationID`, `fare_amount`, `tip_amount`.

Швидкий старт
----------------
1. Зібрати проект:

```powershell
dotnet build ./CrewRedTestTask/CrewRedTestTask.csproj
```

2. Запустити (приклад):

```powershell
dotnet run --project ./CrewRedTestTask/CrewRedTestTask.csproj -- --file "C:\path\to\input.csv" --connection "Server=.;Database=Taxi;User Id=sa;Password=YourPass;"
```

CLI-аргументи
----------------
- `--file` : шлях до вхідного CSV (обов'язково)
- `--connection` : SQL Server connection string (обов'язково)
- `--batch-size` : розмір партії для вставки (опціонально, за замовчуванням — 10_000)
- `--convert-utc` : опція для конвертації часу з EST → UTC (опціонально)

Database скрипти
----------------
- Скрипти для створення БД і таблиці знаходяться в папці `sql`:
	- `sql/create-database.sql`
	- `sql/create-table.sql`

Схема та індекси
----------------
- Типи колонок: `DATETIMEOFFSET` для часу (або `datetime2` з урахуванням UTC), `INT` для `passenger_count`, `DECIMAL(10,2)` для сум та відстані.
- Індекси:
	- Non-clustered index на `PULocationID` (для агрегацій по `tip_amount`).
	- Покриття/індекси на `trip_distance` і `tpep_pickup_datetime` для прискорення TOP-100 запитів.

Обробка даних
----------------
- Парсинг здійснюється потоково (streaming), щоб уникнути завантаження всього файлу в пам'ять.
- Очищення:
	- `store_and_fwd_flag`: `N` → `No`, `Y` → `Yes`.
	- Видалення провідних/приших пробілів у текстових полях.
	- Валідація дат і числових полів; рядки з критичними помилками логуються і пропускаються.
- Дедуплікація: дублікат визначається за комбінацією `pickup_datetime`, `dropoff_datetime`, `passenger_count`. Видалені дублікати записуються у `duplicates.csv`.
- (Опція) Конвертація часової зони: вхідні дані в EST — можна конвертувати в UTC перед вставкою за допомогою `--convert-utc`.

Ефективна масова вставка
----------------
- Використовується `SqlBulkCopy` для швидкої вставки великих обсягів.
- Дані групуються в батчі (`--batch-size`) і вставляються по черзі.
- Рекомендації при великих завантаженнях: тимчасово відключити/відкласти індекси, використовувати `SIMPLE` або `BULK_LOGGED` recovery model для зменшення логування, застосувати table locking для швидшої вставки.

Дублікати
----------------
- Під час обробки формується унікальний ключ (наприклад, хеш рядка `pickup|dropoff|passenger_count`). Якщо ключ вже зустрічався — рядок вважається дублікатом.
- Всі видалені дублікати зберігаються у файлі `duplicates.csv` у виконуваній теці (наприклад, `bin/Debug/net10.0/duplicates.csv`).

Безпека та валідація
----------------
- Обробка небезпечних/недовірених джерел: сувора валідація типів, параметризовані запити при додаткових SQL-операціях, логування помилок.
- Уникаємо SQL-ін'єкцій та обробляємо потенційно зіпсовані рядки без аварійного завершення.

Масштабування для 10GB файлу (коротко)
----------------
- Потоковий парсинг без буферизації всього файлу; читання та обробка чанками.
- Використання producer/consumer (черга) з кількома споживачами для паралельних вставок.
- Дедуплікація в пам'яті замінюється на зовнішнє рішення (Redis, on-disk hash, або попереднє сортування з merge-степом).
- Можливе шардування або розбиття таблиці по партиціях для паралельної вставки.

Deliverables
----------------
- Повний вихідний код проекту в цьому репозиторії (основний проект — `CrewRedTestTask`).
- SQL-скрипти: `sql/create-database.sql`, `sql/create-table.sql`.
- Файл `duplicates.csv` буде згенерований під час виконання.
- Після виконання програми вона виводить кількість вставлених рядків у консоль.

Припущення
----------------
- Вхідні дати в EST (UTC-5/UTC-4 при DST); за замовчуванням зберігаємо як UTC якщо вказано опцію.
- Дублікація визначається строго по `pickup_datetime`, `dropoff_datetime`, `passenger_count`.
- CSV може містити пошкоджені рядки — такі рядки логуються і пропускаються.
- На машині є доступ до SQL Server (локального або віддаленого).
